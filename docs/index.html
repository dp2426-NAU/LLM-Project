Project Overview
What does DocuMind do?
DocuMind solves two real problems that compliance teams and researchers face every day — explained in plain English.
Module I

RegDelta

Regulatory Change Impact Analyzer

Imagine a bank's compliance team hears that the SEC updated a regulation. They need to figure out which of their 200 internal policy documents are now out of date — a process that used to take 2–4 weeks manually.

RegDelta does this in seconds. It reads both versions of the regulation, finds exactly which sections changed, retrieves the policies most likely to be affected, and generates a risk-classified impact report — all automatically.

  • Detects which sections of a regulation were Added, Modified, or Removed
  • Searches your internal policy library using semantic AI (not just keyword search)
  • Asks an LLM: "What changed and why does it matter?" — per section
  • Produces a final report: risk level (LOW/MEDIUM/HIGH/CRITICAL), affected policies, recommended actions
Module II

SynapseIQ

Multi-Agent Research Intelligence Platform

Imagine an analyst who needs to read 10 research papers and write a synthesis report. That could take days. Simple AI chatbots just paste back quotes — they don't think critically.

SynapseIQ uses 4 specialized AI agents working in sequence. One agent reads and extracts findings. A second critiques it. A third writes the synthesis. A fourth scores the quality and issues a PASS or FAIL verdict before the answer is delivered.

  • Agent 1 (Researcher): Extracts key findings from your document corpus
  • Agent 2 (Critic): Finds gaps, contradictions, and unsupported claims
  • Agent 3 (Synthesizer): Writes the final knowledge document
  • Agent 4 (Validator): Scores quality on 4 dimensions and issues a verdict
Architecture
How does it work under the hood?
Both modules use the same core technology — Retrieval-Augmented Generation (RAG) — but apply it differently.
Key Idea: RAG (Retrieval-Augmented Generation)
Instead of asking an AI to answer from memory (which leads to hallucination), DocuMind first retrieves the most relevant text chunks from your documents using vector search, then passes those chunks as evidence to the LLM. The AI can only answer based on what it was given — making outputs factual and traceable.

Step-by-Step: What happens when you upload a document?

📄

1. Document Ingestion

Your PDF, TXT, or MD file is read and split into overlapping text chunks (~800 words each). Each chunk gets a unique SHA-256 fingerprint so re-uploading the same file never creates duplicates.

🧮

2. Embedding

Each chunk is converted into a 384-number mathematical vector using the all-MiniLM-L6-v2 model (runs locally on CPU, no API key needed). Similar sentences produce similar vectors.

🗄️

3. Vector Storage

Vectors are stored in ChromaDB — an AI database that can instantly find the most semantically similar chunks to any query, even if they don't share the same words.

🔍

4. Query & Retrieval

When you ask a question, it's also converted to a vector. ChromaDB finds the top-6 most similar document chunks. These become the factual grounding context for the AI.

🤖

5. LLM Generation

The retrieved chunks + your question are sent to OpenAI's gpt-4o-mini (or local Ollama). The AI reads the evidence and generates a grounded, factual answer — not a hallucination.

✅

6. Quality Gate

SynapseIQ's ValidatorAgent scores the output on 4 dimensions before returning it. A composite score determines PASS / CONDITIONAL_PASS / FAIL. RegDelta classifies risk level.

External Services
Which APIs and services are used?
DocuMind uses a minimal set of external services. The embedding model runs fully locally — only LLM inference requires an external API (and even that can be run locally with Ollama).

OpenAI API Paid · cloud

Used for all LLM inference calls — the "thinking" part of the pipeline. DocuMind defaults to gpt-4o-mini, OpenAI's fastest and most cost-efficient model.

What it's called for:
  • RegDelta: annotating changed regulatory sections (1 call per section) and generating the full impact report (1 call)
  • SynapseIQ: each of the 4 agents makes 1 LLM call — total 4 calls per synthesis request

Cost: gpt-4o-mini costs $0.00015 per 1K input tokens and $0.00060 per 1K output tokens. A typical SynapseIQ synthesis costs ~$0.004–0.007. You can also use GPT-4o for higher quality at higher cost.

How to set it up: Add OPENAI_API_KEY=sk-... to your .env file. Get a key at platform.openai.com/api-keys.

Ollama (Local LLM) Free · local

An alternative to the OpenAI API that runs LLMs entirely on your own machine — completely free, no internet required, no API key needed.

How to switch: Set LLM_PROVIDER=ollama and OLLAMA_MODEL=llama3 in your .env file, then install Ollama from ollama.ai and run ollama pull llama3.

Trade-off: Local inference is slower and produces slightly lower quality than gpt-4o-mini, but costs $0.00 and works offline. Recommended for development and testing.

sentence-transformers / all-MiniLM-L6-v2 Free · local

The embedding model that converts text into 384-dimensional vectors. Downloads automatically on first run (~90MB). Runs entirely on CPU — no GPU, no API key, no cost.

What it does: Converts any piece of text into a list of 384 numbers that capture its meaning. Similar sentences produce similar numbers, which is how ChromaDB can find semantically related chunks.

Why this model: It's the best balance of speed, size, and quality for semantic search on CPU. It produces results that are competitive with much larger models for retrieval tasks.

ChromaDB Free · local

The vector database that stores all embedded document chunks and answers similarity queries. Runs entirely in-process — no separate server, no cloud account.

How it works: ChromaDB uses an HNSW (Hierarchical Navigable Small World) index to find the nearest neighbor vectors to a query in milliseconds, even over millions of documents. Data persists to ./chroma_store/ on disk.

Cost: Completely free and open-source. No API key required.

FastAPI + SQLite Free · local

The REST API framework (FastAPI) and the session database (SQLite) both run locally. No external services needed.

FastAPI: Provides the REST endpoints, automatic OpenAPI/Swagger documentation, Pydantic request validation, and async request handling.

SQLite: Stores all regulation versions, impact reports, pipeline sessions, and agent traces permanently on disk. No database server required.

ABSTRACT LoRA (Low-Rank Adaptation) and full fine-tuning represent two contrasting strategies for adapting pretrained large language models to downstream tasks. Full fine-tuning updates all model parameters, offering maximal expressiveness at high compute cost. LoRA constrains updates to low-rank matrices injected into weight layers, reducing trainable parameters by orders of magnitude. [source: llm_finetuning.txt, chunk_1] MAIN FINDINGS 1. Parameter Efficiency — LoRA reduces trainable parameters by ~10,000× for 7B–70B models. A GPT-3-scale model needs ~4.7M trainable params with LoRA (rank=4) vs. 175B for full fine-tuning. [source: llm_finetuning.txt, chunk_2] 2. Task Performance — On standard benchmarks (GLUE, MMLU), LoRA matches full fine-tuning within 1–3% on most tasks. Gaps widen when the target distribution diverges significantly from pretraining data. [source: transformers_overview.txt, chunk_5] 3. Memory Requirements — Full fine-tuning a 7B model needs ~112 GB GPU memory. LoRA needs ~18 GB, enabling fine-tuning on a single A100 or consumer GPU cluster. The API runs on your machine. Once running, the full Swagger UI is available at http://localhost:8000/api/v1/docs — no frontend needed.

Create virtual environment python -m venv .venv source .venv/bin/activate # Windows: .venv\Scripts\activate # 3. Install dependencies pip install -r requirements.txt # 4. Add your OpenAI key cp .env.example .env # Edit .env: OPENAI_API_KEY=sk-... # 5. Start the server uvicorn backend.main:app --reload # 6. Open in browser # http://localhost:8000/api/v1/docs
